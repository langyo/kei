### ما يضيفه الـ PR

**OSTD (`ostd/src/arch/aarch64/`) :**
- `boot/` — نقطة دخول BSP، جداول صفحات الإقلاع
- `mm/` — جداول صفحات ARM64 (ترقيم من 4 مستويات)، إعداد MMU
- `task/` — تبديل السياق، حفظ/استعادة FPU/SIMD
- `irq/` — متحكّم المقاطعات GICv3 (يستخدم crate من جهة خارجية)
- `timer/` — المؤقّت العام لـ ARM (مادي EL1)
- `trap/` — معالجة استثناءات EL1 (sync/IRQ/FIQ/SError)
- `cpu/` — ميزات الـ CPU، SMP عبر PSCI
- `iommu/` — كعب IOMMU
- `device/` — اكتشاف الأجهزة عبر FDT
- `io/` — تجريد MMIO
- `power.rs` — إدارة الطاقة عبر PSCI (إيقاف/إعادة تشغيل)
- `serial.rs` — وحدة تحكّم UART من نوع PL011

**النواة (`kernel/src/arch/aarch64/`) :**
- دعم العمليات / الخيوط
- جدول استدعاءات النظام (EL0 ← EL1)
- معالجة TLS (TPIDR_EL0)
- تعداد PCI
- دعم VirtIO
- إصلاح خطأ مسح TLB في `KVirtArea::drop()`

**OSDK :**
- صيغة `Image` خام لـ ARM64 لبروتوكول إقلاع Linux في QEMU
- مخطط QEMU Arm64 في `OSDK.toml`
- CI عبر GitHub Actions لفحص + ترجمة arm64

## استراتيجية kei

يدمج kei هذا الفرع عبر git (وليس عبر الرُقع). هذا يعني أن :

1. شجرة `ostd/src/arch/aarch64/` الكاملة موجودة في مستودع kei
2. يمكننا تعديل أي ملف مباشرةً
3. المزامنة مع المصدر المنبع تتم عبر `git merge` وليس `quilt push`
4. عندما يدمج المصدر المنبع في النهاية تطبيقاً مختلفاً لـ arm64،
   نُعيد تأسيس BSP الخاص بنا فوق كود المعمارية الجديد

## المشاكل المعروفة في فرع arm64-support

| المشكلة | الخطورة | إجراء kei |
|---------|---------|-----------|
| كل الكود مُولّد بواسطة LLM | عالية | تدقيق M2: مراجعة كل ملف، إصلاح الشوائب |
| crate خارجي لـ GICv3 | متوسطة | الاستبدال بمُشغّل داخلي |
| اختبارات في QEMU فقط | عالية | الإقلاع على عتاد حقيقي NanoPi R3S |
| لا يوجد SMP/متعدد الأنوية | متوسطة | إضافة تشغيل CPU ثانوي عبر PSCI |
| قديم (متأخر عن الفرع الرئيسي المنبعي) | منخفضة | إعادة تأسيس مزامنة دورية |
| تعليقات مطوّلة بأسلوب LLM | منخفضة | التنظيف أثناء التدقيق |
